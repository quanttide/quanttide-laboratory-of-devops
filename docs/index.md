# 量潮 DevOps 实验：核心矛盾与方向

## 问题

CLI 已明显进入瓶颈。根本原因是：

> **工具在设计层面假设使用者会遵守流程，但 AI 的使用模式恰恰是不遵守流程。**

AI 绕过流程的记录（`data/insight`）揭示了一个重复模式——AI 认为自己的判断比工具的流程检查更可靠。每一次绕过都是"我觉得这次不需要"的决策，每次都被用户证明是错的。但由于会话没有记忆，同样的绕过会重复犯。

## 方向：从"约束行为"到"约束结果"

早期思路是约束 AI 的行为——要求 AI 按步骤执行。这很难，而且每次对话都要重新教。

现在的思路是**约束结果**——不管 AI 怎么跳步、怎么绕过，后台以一个高频收敛循环扫描所有 artifact 的一致性，发现问题按规则自动修复。这就把一个不可控的流程问题，变成了一个可控的状态机收敛问题。

```
AI → 任意操作（包括绕过） → 状态变化
     ↕                          ↕
  高频收敛循环 ← 扫描 artifact 一致性 ← 
  发现问题 → 自动修复 → 状态收敛
```

## 核心模型：Tag-CHANGELOG-Release 三角

每个发布的完整性由三个 artifact 共同决定：

| Artifact | 性质 | 可修复性 |
|----------|------|---------|
| **Tag** | 事实源 | 不可自动修复，需人类确认 |
| **CHANGELOG** | 规范事实源 | 可从 git log 补写 |
| **GitHub Release** | 派生制品 | 可从 CHANGELOG 重建 |

Audit 的职责不是"检查每个 artifact 是否存在"，而是**检查三者之间是否一致**——把因果约束关系模型化，发现断裂后按规则自动收敛。

## 实验室在这个上下文中的角色

实验室（`examples/default`）是用来验证这些想法的试验场：

1. **CLI 命令原型的端到端验证**——plan/audit/publish 等命令能否正确执行
2. **绕过场景的模拟**——手动破坏 artifact 一致性，验证 provider 能否发现并修复
3. **架构决策的验证**——gix/git2/CLI 分工、devops-vs-code 边界、本地-vs-云端分界线
