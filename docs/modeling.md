# 发布建模：Artifact 三角与状态收敛

## Artifact 三角

每个发布的完整性由三个 artifact 共同决定，它们之间具有因果链：

```
CHANGELOG ← 规范事实源，派生制品
    ↓
Tag ← 事实源，不可移动
    ↓
GitHub Release ← 派生制品，可重建
```

`publish` 命令原本按此顺序创建三者：CHANGELOG → tag → Release。但 AI 可以绕过 `publish` 独立操作其中任何一个，打破因果关系。

## 状态空间

扫描结果为三者状态的组合，共 2³ = 8 种状态：

| CHANGELOG | Tag | Release | 判定 | 处理 |
|-----------|-----|---------|------|------|
| ✅ | ✅ | ✅ | **正常** | 无操作 |
| ❌ | ✅ | ✅ | 缺 CHANGELOG | 从 git log 补写 |
| ✅ | ✅ | ❌ | 缺 Release | `gh release create` 从 CHANGELOG 补 |
| ✅ | ❌ | ❌ | 未发布 | 无操作（正常未发布状态） |
| ❌ | ✅ | ❌ | 只有 tag | 缺 CHANGELOG + Release |
| ❌ | ❌ | ✅ | **不可能** | Release 依赖 tag 创建 |
| ✅ | ❌ | ✅ | **不可能** | Release 依赖 tag 创建 |
| ❌ | ❌ | ❌ | 无 artifact | 无操作 |

## 反脆弱修复策略

| 问题 | 修复方式 | 原则 |
|------|---------|------|
| 缺 CHANGELOG | 从 git log 补写 | 修改 tag 会改变事实源，需谨慎 |
| 缺 Release | `gh release create` 补充 | 不改变任何事实源，安全 |
| 缺 tag | 不可自动修复 | tag 是事实源，需人类确认 |
| CHANGELOG 内容过时 | 从当前 tag~HEAD 的 git log 更新 | 增量追加 |

## 从审计到收敛

传统 audit 的逻辑是"发现问题后报告"。反脆弱 audit 的逻辑是"发现问题后自动收敛到正确状态"。

关键转变：

```
传统: 扫描 → 发现断裂 → 报告人类 → (等待) → 人类修复
反脆弱: 扫描 → 发现断裂 → 按规则自动修复 → 状态收敛
       ↓ 无法自动修复的 → 标记搁置 → 人类决策
```

## 状态机视角

整个过程可以建模为一个状态机：

1. **初始状态**：任意 artifact 组合（AI 操作后的混乱状态）
2. **扫描**：读取当前 Tag/CHANGELOG/Release 状态
3. **判定**：对照状态空间表，确定当前处于哪个状态
4. **修复**：对可自动修复的状态执行修复动作
5. **收敛**：所有 scope 回到 ✅✅✅ 状态

服务端（Provider）的作用就是以极高的频率运行这个状态机。
