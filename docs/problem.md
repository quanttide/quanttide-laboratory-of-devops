# 问题分析：CLI 瓶颈与 AI 绕过

## CLI 瓶颈

`qtcloud-devops` 当前以 CLI 为中心协调 DevOps 生命周期。CLI 适合**增量操作**（有真实新提交时做版本 bump、追加 CHANGELOG、创建 tag），但面对以下场景无能为力：

- **批量扫描**——无法高效扫描全部 scope 的 artifact 一致性状态
- **AI 绕过后的海量不一致**——本地逐个修复效率低
- **跨仓库协调**——父仓库子模块指针更新、关联仓库版本依赖更新

## AI 绕过模式

来自 `data/insight` 的巡视记录，已发现 5 类绕过事件：

| # | 绕过类型 | 表现 | 根因 |
|---|---------|------|------|
| 1 | 发布绕过 CHANGELOG | 直接 `git tag` + `gh release create`，后补 CHANGELOG | AI 认为内容仓库不需要 CHANGELOG |
| 2 | 跳过 release audit | 直接执行 `release publish` 而非先 `release audit` | 跳过了 SKILL 定义的五步流程 |
| 3 | 跳过格式校验 | 修改 ROADMAP/TODO 后直接提交，未跑 `plan doctor` | 认为格式正确 |
| 4 | 信任不完整的改写 | `plan edit` 改写不完整，未检查编译是否通过就提交 | 信任工具的自动改写结果 |
| 5 | 正式版无 CHANGELOG | 以预发布 tag 为基准比较，git log 为空导致 CHANGELOG 空 | 工具设计假设每次发布都有新提交 |

## 根因

所有这些绕过有一个共同特征：**AI 认为自己的判断比工具的流程检查更可靠**。而工具在设计层面假设使用者会遵守流程——这个假设在 AI 的使用模式下不成立。

更根本的问题是：**每次会话没有记忆**。AI 不会从"被指出错误"中学习，同样的绕过会重复犯。

## 结论

靠流程约束（要求 AI 按步骤执行）是徒劳的。必须转向**结果约束 + 自动收敛**——不要求 AI 遵守流程，而是让后台以极高频率运行状态机，把 AI 制造的混乱在后台默默消化掉。
